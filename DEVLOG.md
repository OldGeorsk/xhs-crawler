# Development Log

## 采集层优化

### 1. 卡片提取：从单次提取到边滚边采

**问题**：XHS 创作者主页使用虚拟滚动（virtual scrolling），DOM 中的卡片会随着滚动被回收。如果只在滚动结束后提取，最新发布的内容（位于页面顶部）已被移除，导致只能采集到末尾的一小批旧笔记。

**现象**：慢慢Daily 主页实际有 30 条笔记，旧逻辑只采到 9 条，缺少全部 2026 年的新内容。

**方案**：将分离的 `_scroll_to_load` + `_extract_notes` 合并为 `_scroll_and_collect`，**每滚动一步就扫描当前 DOM 中的所有卡片**，累积去重。这样无论虚拟滚动如何回收 DOM 节点，每个滚动位置出现的笔记都能被捕获。

```
旧逻辑:  滚动到底 → 提取一次   → 9 条 (只剩底部)
新逻辑:  滚一步→采→滚一步→采→... → 30 条 (全覆盖)
```

### 2. 详情页采集：xsec_token 链路

**问题**：直接 `page.goto("/explore/{note_id}")` 被 XHS 返回 461（反爬拦截）。必须带上 `xsec_token` 且从主页卡片点击进入才被信任。

**方案**：在主页采集阶段提取每个卡片的 `detail_url`（封面链接中的完整 URL，含 `xsec_token`），进入详情页时使用该完整 URL。模拟了"从主页点进笔记"的自然浏览行为。

### 3. 多布局兼容：新版 vs 旧版创作者主页

**问题**：XHS 有两套创作者主页架构：
- **旧版**：note ID 在 `<a href="/explore/{note_id}">` 中，可直接提取
- **新版**：SSR HTML 中 `noteId: ""`（空字符串），note ID 由 JS 动态加载

旧版在 Playwright 独立浏览器中即可采集，新版则需要更强的认证态。

**方案**：采集器通过 `a[href*="/explore/"]` 提取 note ID，同时兼容新旧两版 DOM 结构。新版页面的认证问题由 CDP 方案解决（见下方）。

## 认证层优化

### 1. cookies.json → storage_state

**问题**：Cookie-Editor 导出的纯 cookie 文件不稳定，几小时就可能过期，且缺乏 localStorage / IndexedDB 等浏览器指纹。

**方案**：改用 Playwright 的 `storage_state`，保存完整浏览器状态（cookie + localStorage + IndexedDB + Service Worker）。同时提供 `login_helper.py` 扫码登录工具。

### 2. storage_state → CDP（连接真实 Chrome）

**问题**：storage_state 对旧版主页足够，但新版主页要求更强的认证态。即使 storage_state 有效，新版主页仍可能弹出登录遮罩，导致卡片无法点击、note ID 无法获取。

**现象**：安安不困（新版布局）用 storage_state 采集到 0 条，切换到 CDP 后采集到 50 条。

**方案**：通过 Chrome DevTools Protocol 连接用户日常使用的真实 Chrome 浏览器。用户只需启动 Chrome 时加 `--remote-debugging-port=9222` 参数，Playwright 复用其真实登录态。

```
storage_state:  旧版 50 条 ✅  |  新版 0 条 ❌
CDP:            旧版 50 条 ✅  |  新版 50 条 ✅
```

CDP 已是默认模式（`python main.py`），不可用时自动回退 storage_state。

## 同步策略优化

### 1. 增量同步：从"遇已知即停"到"全量扫描跳过已知"

**问题**：旧策略假设主页卡片按时间倒序排列。但 XHS 的卡片排列并非严格时间序，遇第一条已知笔记就停止可能漏掉后面的新笔记。

**方案**：改为遍历所有可见卡片，跳过数据库中已有的，只下载未知的。

### 2. 多创作者管理 → 单创作者每次运行

**问题**：连续同步多个创作者会导致浏览器会话退化，第二个创作者时常遇到登录弹窗。

**方案**：管理（`--add` / `--remove` / `--list`）和采集解耦。每次 `python main.py` 只同步一个创作者（默认第一个启用的，`--creator` 指定），保持会话新鲜。

## 人类行为模拟

所有自动操作插入随机延时和随机滚动量，降低被 XHS 风控标记的概率：

| 操作 | 延时 |
|---|---|
| 主页加载后 | 2500ms ± 1000ms |
| 卡片间滚动 | 1800ms ± 600ms |
| 滚动步长 | 400~900px 随机 |
| 点进详情前 | 1500ms ± 500ms |
| 详情页停留 | 3500ms ± 1500ms |
| 返回主页后 | 2000ms ± 800ms |

### 3. 笔记间发呆间隔

**问题**：每篇笔记都无缝衔接、逐条点进详情页，阅读时间恒定 3~5 秒，行为模式过于机械，容易被 XHS 风控标记为自动化脚本。

**现象**：同步 8 个博主（322 条笔记）后，XHS 弹出"账号存在疑似使用脚本自动浏览"的警告。

**方案**：在每两条笔记的处理之间插入 `random.randint(15, 25)` 秒的随机发呆间隔，模拟真人看一篇、滑一滑、走神一会再看下一篇的行为节奏。

```
之前:  笔记1 → 笔记2 → 笔记3 → ...  (连续, 反爬一眼假)
现在:  笔记1 → [发呆18s] → 笔记2 → [发呆22s] → 笔记3 → [发呆15s] → ...
```

### 4. 随机点赞行为

**问题**：真人浏览小红书时会偶尔点赞感兴趣的内容。如果采集了上百条笔记却一次都不点赞，也是异常信号。

**方案**：每处理 2~5 条笔记时，随机为下一篇笔记点击点赞按钮。点赞前会检查 `like-wrapper` 是否已有 `like-active` 类（避免重复点赞或取消点赞）。

```
笔记3 → 点赞 ❤️   (间隔3条)
笔记7 → 点赞 ❤️   (间隔4条)
笔记11 → 点赞 ❤️  (间隔4条)
...
```
